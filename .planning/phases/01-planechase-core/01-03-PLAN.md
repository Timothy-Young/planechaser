   - Sign In / Create Account button (shadcn Button): full width, height 48px, background var(--color-accent), Russo One 16px, white text. Button is `disabled` during loading.
   - OR divider: 1px var(--color-border) with "OR" Chakra Petch 14px var(--color-text-muted) centered
   - "Continue with Google" button (shadcn Button variant="outline"): full width, height 48px, background var(--color-surface) with border var(--color-border), Chakra Petch 14px 600, white text, Lucide `Chrome` icon or a simple "G" SVG left-aligned

State management (all local useState):
- `mode: 'signin' | 'signup'` — toggles between sign-in and account creation
- `email: string`
- `password: string`
- `loading: boolean`
- `error: string | null`

On Sign In (mode='signin'): call `createClient().auth.signInWithPassword({ email, password })`. On error, set error = "Sign in failed. Check your email and password, then try again." (exact copy from UI-SPEC copywriting contract). On success, router.push('/setup').

On Create Account (mode='signup'): call `createClient().auth.signUp({ email, password, options: { data: { display_name: email.split('@')[0] } } })`. On signup success, also create a profile row in the `profiles` table: `createClient().from('profiles').insert({ id: user.id, display_name: email.split('@')[0] })`. Then router.push('/setup').

On Google: call `createClient().auth.signInWithOAuth({ provider: 'google', options: { redirectTo: `${window.location.origin}/auth/callback` } })`. The OAuth redirect is handled by Supabase — no callback route needed for Phase 1 since Supabase handles the session creation. After OAuth, the middleware will redirect the user to /setup.

Inline error display: render error string in Chakra Petch 14px var(--color-destructive) below the main button, inside the form card. Clear error when the user starts typing again.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser form → Supabase Auth | Email/password cross this boundary; Supabase handles hashing |
| Browser → Google OAuth | Redirect to Google; Supabase handles token exchange |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | src/app/(auth)/auth/page.tsx sign-in | mitigate | Supabase Auth handles bcrypt comparison; app never sees raw passwords server-side |
| T-03-02 | Tampering | WotC disclaimer sessionStorage flag | accept | Clearing sessionStorage only hides/re-shows the disclaimer — no security impact |
| T-03-03 | Information Disclosure | Auth error messages | mitigate | Error message is generic ("Sign in failed...") — does not confirm whether email exists |
| T-03-04 | Elevation | /setup route post-auth | mitigate | Middleware re-validates session server-side; client-side router.push('/setup') alone does not grant access |
| T-03-05 | Tampering | profiles.display_name on signup | mitigate | display_name derived from email prefix on signup; user can update via Profile screen (Phase 4); UNIQUE constraint prevents duplicates |
</threat_model>
